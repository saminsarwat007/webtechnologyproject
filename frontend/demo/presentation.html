What My Modules Do (In Simple Words)

M2 Job Management

This module lets admins post job listings for students to browse. Think of it like a job board.

  • Admins can create new job posts (title, company, type, deadline, description)
  • Admins can edit existing job posts
  • Admins can deactivate a job (soft delete — hides it from students but keeps the data)
  • Students can browse all active jobs, search by name, and filter by type
  • Students can view a full detail page for each job

In short: admins manage the job board, students browse it.

M3 Application Tracking

This module lets students apply for jobs and admins track those applications. Think of it like an application tracker.

  • Students can apply to a job with an optional cover letter
  • A student cannot apply twice to the same job (blocked by the system)
  • Students can see all their applications and the status of each one
  • Students can withdraw an application — but only if it's still "pending"
  • Admins can see every application in the system and change the status (pending → reviewed → accepted/rejected)

In short: students apply and track, admins review and decide.

In simple terms: Module 2 is a job board — admins post jobs, students browse them. Module 3 is an application tracker — students apply for jobs and can see their status, admins review applications and accept or reject them. A student can't apply to the same job twice, and can only withdraw while the application is still pending. Once an admin reviews it, the student can't withdraw anymore.

Database Design — My Tables

jobs table

CREATE TABLE jobs (
  job_id INT AUTO_INCREMENT PRIMARY KEY,
  company_id INT NOT NULL,
  posted_by  INT NOT NULL,
  title       VARCHAR(150) NOT NULL,
  type ENUM('internship','fulltime',
             'parttime') NOT NULL,
  description TEXT NOT NULL,
  requirements TEXT,
  deadline DATE NOT NULL,
  is_active TINYINT(1) DEFAULT 1,
  created_at TIMESTAMP DEFAULT
    CURRENT_TIMESTAMP,
  FOREIGN KEY (company_id)
    REFERENCES companies(company_id),
  FOREIGN KEY (posted_by)
    REFERENCES users(user_id)
);

applications table

CREATE TABLE applications (
  application_id INT AUTO_INCREMENT
    PRIMARY KEY,
  job_id  INT NOT NULL,
  user_id INT NOT NULL,
  cover_letter TEXT,
  status ENUM('pending','reviewed',
    'accepted','rejected')
    DEFAULT 'pending',
  applied_at TIMESTAMP DEFAULT
    CURRENT_TIMESTAMP,
  FOREIGN KEY (job_id)
    REFERENCES jobs(job_id),
  FOREIGN KEY (user_id)
    REFERENCES users(user_id),
  UNIQUE KEY uniq_job_user
    (job_id, user_id)
);

UNIQUE (job_id, user_id) — a student cannot apply to the same job twice. Foreign keys link jobs → companies and jobs → users (who posted it).

These are my two database tables. The jobs table has foreign keys to companies and users — each job belongs to a company and was posted by a user. The applications table has a unique constraint on (job_id, user_id) which prevents a student from applying to the same job twice at the database level. The status field uses an ENUM type with four values: pending, reviewed, accepted, rejected.

Application Submitted!

Backend Code — Apply (POST)

public function create(Request $request, Response $response): Response
{
    $payload = (array) $request->getAttribute('jwt_payload');
    $userId  = (int) ($payload['user_id'] ?? 0);
    $body    = $this->readJson($request);
    $jobId   = (int) ($body['job_id'] ?? 0);

    // 1. Validate input
    if ($jobId <= 0) {
        return Json::write($response, 400, [
            'success' => false,
            'message' => 'Validation failed.',
            'errors'  => ['job_id' => 'job_id is required.'],
        ]);
    }

    // 2. Check job exists and is active
    $stmt = $pdo->prepare('SELECT job_id, is_active FROM jobs WHERE job_id = :id');
    $stmt->execute([':id' => $jobId]);
    $job = $stmt->fetch(PDO::FETCH_ASSOC);
    if (!$job) return Json::write($response, 404, [...]);
    if ((int) $job['is_active'] !== 1) return Json::write($response, 400, [...]);

    // 3. Check not already applied (prevent duplicates)
    $stmt = $pdo->prepare(
        'SELECT application_id FROM applications
         WHERE job_id = :jid AND user_id = :uid LIMIT 1'
    );
    $stmt->execute([':jid' => $jobId, ':uid' => $userId]);
    if ($stmt->fetch()) {
        return Json::write($response, 409, [
            'success' => false,
            'message' => 'You have already applied for this job.',
        ]);
    }

    // 4. Insert application with PDO prepared statement
    $stmt = $pdo->prepare(
        'INSERT INTO applications (job_id, user_id, cover_letter, status)
         VALUES (:jid, :uid, :cl, "pending")'
    );
    $stmt->execute([
        ':jid' => $jobId, ':uid' => $userId,
        ':cl'  => $coverLetter !== '' ? $coverLetter : null,
    ]);

    return Json::write($response, 201, [
        'success' => true,
        'message' => 'Application submitted.',
    ]);
}
This is the backend code for applying to a job. Step 1: validate the job_id. Step 2: check the job exists and is active using a prepared statement. Step 3: check if the student already applied — if yes, return 409 Conflict. Step 4: insert the application with PDO prepared statements. All user input goes through named placeholders, which prevents SQL injection.

Withdraw — Ownership & Status Check

public function delete(Request $request, Response $response, array $args): Response
{
    $id      = (int) ($args['id'] ?? 0);
    $payload = (array) $request->getAttribute('jwt_payload');
    $userId  = (int) ($payload['user_id'] ?? 0);

    $stmt = $pdo->prepare(
        'SELECT user_id, status FROM applications
         WHERE application_id = :id LIMIT 1'
    );
    $stmt->execute([':id' => $id]);
    $row = $stmt->fetch(PDO::FETCH_ASSOC);

    // 1. Not found → 404
    if (!$row) return Json::write($response, 404, [...]);

    // 2. Not your application → 403 Forbidden
    if ((int) $row['user_id'] !== $userId) {
        return Json::write($response, 403, [
            'success' => false,
            'message' => 'You can only withdraw your own applications.',
        ]);
    }

    // 3. Already processed → 400 (only pending can be withdrawn)
    if ($row['status'] !== 'pending') {
        return Json::write($response, 400, [
            'success' => false,
            'message' => 'Only pending applications can be withdrawn.',
        ]);
    }

    // 4. Hard delete from database
    $stmt = $pdo->prepare('DELETE FROM applications WHERE application_id = :id');
    $stmt->execute([':id' => $id]);

    return Json::write($response, 200, [
        'success' => true,
        'message' => 'Application withdrawn.',
    ]);
}
The withdraw endpoint has three security checks: 1) Does the application exist? (404) 2) Does it belong to this user? (403 Forbidden if not) 3) Is it still pending? (400 if already reviewed/accepted/rejected). Only then is it hard-deleted from the database. This prevents students from withdrawing applications that have already been processed by an admin.

Backend Code — Create Job (POST)

public function create(Request $request, Response $response): Response
{
    $body    = $this->readJson($request);
    $payload = (array) $request->getAttribute('jwt_payload');
    
    // Validate all fields
    $errors = $this->validate($body);
    if (!empty($errors)) {
        return Json::write($response, 400, [
            'success' => false,
            'message' => 'Validation failed.',
            'errors'  => $errors,
        ]);
    }

    // Insert using PDO prepared statements (SQL injection safe)
    $pdo = Database::getConnection();
    $stmt = $pdo->prepare(
        'INSERT INTO jobs (company_id, posted_by, title, type,
             description, requirements, deadline, is_active)
         VALUES (:cid, :uid, :title, :type,
             :descr, :req, :deadline, 1)'
    );
    $stmt->execute([
        ':cid'      => (int) $body['company_id'],
        ':uid'      => (int) $payload['user_id'],
        ':title'    => $body['title'],
        ':type'     => $body['type'],
        ':descr'    => $body['description'],
        ':req'      => $body['requirements'] ?? null,
        ':deadline' => $body['deadline'],
    ]);
    $id = (int) $pdo->lastInsertId();

    return Json::write($response, 201, [
        'success' => true,
        'message' => 'Job created.',
        'data'    => $row,
    ]);
}
This is the backend code for creating a job. First, validate all fields using the validate() method. Then insert using PDO prepared statements with named placeholders. The posted_by field comes from the JWT payload, not from the request body — so the system always knows who created the job. Returns 201 Created with the new job data.

Soft Delete — Jobs Aren't Really Deleted

public function delete(Request $request, Response $response, array $args): Response
{
    $id = (int) ($args['id'] ?? 0);

    // Check job exists
    $stmt = $pdo->prepare('SELECT job_id FROM jobs WHERE job_id = :id');
    $stmt->execute([':id' => $id]);
    if (!$stmt->fetch()) {
        return Json::write($response, 404, [
            'success' => false,
            'message' => 'Job not found.',
        ]);
    }

    // Soft delete: set is_active = 0 (NOT a real DELETE)
    $stmt = $pdo->prepare(
        'UPDATE jobs SET is_active = 0 WHERE job_id = :id'
    );
    $stmt->execute([':id' => $id]);

    return Json::write($response, 200, [
        'success' => true,
        'message' => 'Job deactivated.',
    ]);
}
Why soft delete? The job record stays in the database for audit. Students just don't see it anymore (index query filters WHERE is_active = 1).
I use soft delete for jobs instead of hard delete. The job record stays in the database with is_active set to 0. This preserves data integrity — existing applications still reference the job. The Browse Jobs page only shows jobs WHERE is_active = 1, so deactivated jobs are hidden from students but the data is preserved for audit.

Backend Code — Update Status (PUT)

public function updateStatus(Request $request, Response $response, array $args): Response
{
    $id     = (int) ($args['id'] ?? 0);
    $body   = $this->readJson($request);
    $status = trim((string) ($body['status'] ?? ''));

    // Validate status value
    if (!in_array($status, ['pending', 'reviewed',
            'accepted', 'rejected'], true)) {
        return Json::write($response, 400, [
            'success' => false,
            'message' => 'Validation failed.',
            'errors'  => ['status' =>
              'Status must be pending, reviewed, accepted or rejected.'],
        ]);
    }

    // Check application exists
    $stmt = $pdo->prepare(
        'SELECT application_id FROM applications
         WHERE application_id = :id LIMIT 1'
    );
    $stmt->execute([':id' => $id]);
    if (!$stmt->fetch()) {
        return Json::write($response, 404, [
            'success' => false,
            'message' => 'Application not found.',
        ]);
    }

    // Update status
    $stmt = $pdo->prepare(
        'UPDATE applications SET status = :s
         WHERE application_id = :id'
    );
    $stmt->execute([':s' => $status, ':id' => $id]);

    return Json::write($response, 200, [
        'success' => true,
        'message' => 'Status updated.',
        'data'    => ['application_id' => $id, 'status' => $status],
    ]);
}
The update status endpoint validates that the status is one of the four allowed values, checks the application exists, then updates it. Only admins and super admins can access this endpoint — enforced by JWT middleware in the route definition.

Frontend Code — Optimistic Update

// ManageApplications.vue — change status with optimistic update
async function changeStatus(app, newStatus) {
  if (newStatus === app.status) return
  const previous = app.status
  
  // 1. Optimistic: update UI immediately
  const idx = applications.value.findIndex(
    a => a.application_id === app.application_id
  )
  if (idx !== -1)
    applications.value[idx] = { ...app, status: newStatus }

  try {
    // 2. Send API request
    await api.put(
      `/applications/${app.application_id}/status`,
      { status: newStatus }
    )
    toast.success(`Status set to ${newStatus}.`)
  } catch (err) {
    // 3. Rollback on error
    if (idx !== -1)
      applications.value[idx] = { ...app, status: previous }
    toast.error('Could not update status.')
  }
}

User sees the change instantly. If the API fails, UI rolls back. This makes the app feel fast.

This is the optimistic update pattern in ManageApplications.vue. Step 1: update the UI immediately. Step 2: send the API request. Step 3: if it fails, roll back to the previous status. This makes the app feel fast and responsive — the user doesn't have to wait for the server to see the change.

Security Features in My Modules

Backend Security

  • PDO Prepared Statements — all queries use :placeholder syntax, preventing SQL injection
  • JWT Role Gating — JwtMiddleware(['admin','superadmin']) on create/update/delete
  • Ownership checks — students can only withdraw their own applications
  • Status checks — only pending applications can be withdrawn
  • Soft delete — jobs are deactivated, not erased (audit trail)
  • Duplicate prevention — unique constraint on (job_id, user_id)

Frontend Security

  • Route guards — router checks role before showing page
  • Auto-logout on 401 — Axios interceptor clears token
  • Client-side validation — all forms validated before API call
  • Confirm dialogs — before destructive actions (delete, withdraw)
  • JWT in localStorage — attached to every request via interceptor
Security is implemented on both sides. Backend: PDO prepared statements prevent SQL injection, JWT middleware enforces role-based access, ownership checks prevent students from accessing other students' data. Frontend: route guards block unauthorized page access, Axios interceptor auto-logs out on 401, confirm dialogs prevent accidental destructive actions.

Input Validation — Both Sides

Frontend (Vue.js)

// ManageJobs.vue validation
function validate() {
  if (!form.company_id)
    errors.company_id = 'Company is required.'
  if (!form.title.trim())
    errors.title = 'Title is required.'
  else if (form.title.length > 150)
    errors.title = 'Title must be 150 chars or fewer.'
  if (!form.description.trim())
    errors.description = 'Description is required.'
  if (!form.deadline)
    errors.deadline = 'Deadline is required.'
  else if (form.deadline < todayStr.value)
    errors.deadline = 'Deadline must be in the future.'
  return Object.values(errors)
    .every(v => v === '')
}

Backend (PHP)

// JobController::validate()
private function validate(array $body): array
{
    $errors = [];
    if (empty($body['company_id']))
        $errors['company_id'] = 'Required.';
    if (empty($body['title']))
        $errors['title'] = 'Required.';
    elseif (strlen($body['title']) > 150)
        $errors['title'] = 'Max 150 chars.';
    if (!in_array($body['type'] ?? '',
        ['internship','fulltime','parttime'], true))
        $errors['type'] = 'Invalid type.';
    if (empty($body['description']))
        $errors['description'] = 'Required.';
    if (empty($body['deadline']))
        $errors['deadline'] = 'Required.';
    return $errors;
}
Validation happens on both sides. The frontend validates for instant user feedback — no need to wait for the server. The backend validates for security — because someone could bypass frontend validation by sending requests directly to the API. Both check required fields, max lengths, valid enum values, and future deadline dates.

HTTP Status Codes I Use

200
OK — GET, PUT, DELETE success
201
Created — POST success
400
Bad Request — validation failed
401
Unauthorized — bad/missing JWT
403
Forbidden — wrong role or not owner
404
Not Found — job/application missing
409
Conflict — already applied / duplicate
I use proper HTTP status codes throughout. 200 for successful reads/updates, 201 for creation, 400 for validation errors, 401 for missing/invalid JWT, 403 for wrong role or not-owner, 404 for not found, and 409 for conflicts like duplicate applications. These standard codes tell the frontend exactly what happened.
